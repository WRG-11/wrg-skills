## 2. Methodology

The audit applies the [`mcp-audit` skill](../../skills/mcp-audit/SKILL.md): five axes scored per tool (discoverability, return-shape, naming, error handling, test coverage), plus decay-candidate detection across four signals (CHANGELOG / README / consumer-import / test).

Severity bucketing uses the default rubric in [`skills/mcp-audit/SEVERITY.md`](../../skills/mcp-audit/SEVERITY.md). One override is called out (F-001: High → Low).

Disclosure handling follows [`docs/disclosure-sop.md`](../disclosure-sop.md). All findings resolve to **public** routing (Low/Info). F-001 was initially flagged for coordinator review; after pickaxe verification (see §3 F-001 correction log), severity was revised down to Low and disclosure routed to a comment on existing upstream issue [#2317](https://github.com/modelcontextprotocol/servers/issues/2317) rather than a new GHSA or issue.

**Honest-scoring discipline**: this is a 1-tool server. Three of the five audit axes (return-shape consistency, naming consistency, decay detection) are trivially satisfied when there is only one tool. The audit still produced value through the test-coverage axis and through the SSRF posture review, which surfaced a maintainer-tracked enhancement gap.

**Attribution discipline**: during initial analysis, a cross-branch `git diff` between an unmerged feature branch (`a1226d4` on `claude/issue-2317-...`) and a mainline commit (`ec20ee7`) was misinterpreted as a regression within main's history. Pickaxe verification (`git log -S "blocked_ip_ranges" -- src/fetch/` returning zero results, `git branch -a --contains a1226d4` showing only the feature branch) corrected this. The lesson: "X commit silently removed Y" claims must always be verified with `git log -S` and `git branch --contains` before reporting. This correction is logged in §3 F-001 and will be incorporated into the skill's audit procedure.

**Framework adaptation**: this server uses the low-level `mcp.server.Server` API, not FastMCP's `@mcp.tool()` decorators. Tool registration is via `@server.list_tools()` + `@server.call_tool()` handler pattern. The skill's grep command (`@mcp.tool()`) does not match; the inventory was done by reading the source directly. The 5-axis rubric applies without further adaptation.

**Skill threshold check**: SKILL.md says "Don't trigger when: Server has <10 tools." With 1 tool, the full 5-axis rubric is overkill. However, the SSRF posture finding (F-001) and the test-coverage gap (F-002) would not have been found without running the audit. Recommendation: for servers with <10 tools, run a lightweight "security + test scan" variant rather than skipping entirely.

### F-001: SSRF posture — no IP/host allowlist (Low)

**Category**: SEC-004 (Arbitrary URL fetch with no allowlist — SSRF)

**Default severity**: High

**Assigned severity**: Low — `OVERRIDE: de-escalated`

**Why this bucket**: De-escalated from High to Low because: (a) the maintainer is aware — issue [#2317](https://github.com/modelcontextprotocol/servers/issues/2317) ("Add support for host allowlisting to Fetch server") has been open since 2025-07-10, labelled `enhancement` (not security/bug); (b) README contains an explicit `[!CAUTION]` block warning users at install time; (c) exploitation requires the MCP client to be on a network with reachable internal services; (d) the server respects `robots.txt` by default, which mitigates some automated abuse.

**What**: The `fetch` tool accepts arbitrary URLs via the `url` parameter and issues HTTP GET requests through httpx without IP address validation or host allowlisting. An MCP client on a network with internal services (cloud metadata endpoints, internal APIs) could be instructed to fetch those resources. This is inherent to the tool's purpose (fetching arbitrary URLs for an LLM) and explicitly documented as a known limitation.

**Context**: An unmerged feature branch (`a1226d4` on `claude/issue-2317-20250817-1707`) attempted to add CIDR-based IP blocking + host allowlisting + CLI args. This branch was never landed to main — `git branch -a --contains a1226d4` shows only the feature branch; `git log -S "blocked_ip_ranges" -- src/fetch/` returns zero mainline results. The reasons it was not merged are unknown to this audit and may reflect an intentional design choice by the maintainer. The feature branch is noted here as prior art, not as evidence of regression.

**Correction log**: The initial version of this finding (pre-review) incorrectly claimed that commit `ec20ee7` "silently removed" the IP blocking code added by `a1226d4`. This was based on a cross-branch `git diff` that compared an unmerged feature branch to a mainline commit, misinterpreted as a regression within main's history. Pickaxe verification (`git log -S`, `git branch --contains`) corrected the attribution. The "regression" framing was wrong; the correct framing is "documented limitation with maintainer-tracked enhancement."

**Disclosure**: Comment on existing issue [#2317](https://github.com/modelcontextprotocol/servers/issues/2317) with audit synthesis. No new issue needed (would duplicate #2317). No GHSA (maintainer labels this `enhancement`, not security vulnerability).

### F-003: Missing return-shape documentation (Info)

**Category**: DISC-004

The `fetch` tool description says: "Fetches a URL from the internet and optionally extracts its contents as markdown." It does not describe what the response looks like (a `TextContent` with `"Contents of {url}:\n{content}"` prefix, with truncation markers and pagination instructions).

An agent calling `fetch` cannot predict:
- That the response has a `"Contents of {url}:"` prefix
- That truncated content ends with `<error>Content truncated. Call the fetch tool with a start_index of N...</error>`
- That `start_index` >= content length returns `<error>No more content available.</error>`

This is cosmetic (Info) because the tool works correctly and the `<error>` markers are self-documenting at runtime.

## 6. Reusable patterns

### Pattern: pickaxe verification for cross-branch claims

When `git log --all` surfaces a security-related commit, the auditor must verify whether it reached mainline before claiming a regression.

**Anti-pattern** (what went wrong in this audit's first draft): `git diff <feature-branch-commit>..<mainline-commit>` shows code "disappearing" because the branches diverged — not because one removed the other's work. This was misinterpreted as "silently removed."

**Correct verification sequence**:
1. `git branch -a --contains <commit>` — did the commit reach main?
2. `git log -S "<symbol>" -- <path>` — was the symbol ever added/removed on the default branch?
3. Only if both confirm mainline presence → claim regression.

This pattern prevented a false Medium finding from reaching upstream disclosure. The lesson is incorporated into the audit methodology: cross-branch diffs are not evidence of mainline regressions.

### Pattern: low-level Server API adaptation

When the target uses `mcp.server.Server` (not FastMCP), the inventory grep changes:

```python
# FastMCP pattern (SKILL.md default)
grep -rn "@mcp.tool()" src/

# Low-level Server API pattern
grep -rn "@server.call_tool\|@server.list_tools\|@server.get_prompt\|@server.list_prompts" src/
```

The 5-axis rubric applies without further changes. The only difference is that tool metadata (name, description, inputSchema) is defined inside the `list_tools` handler rather than via decorators.
