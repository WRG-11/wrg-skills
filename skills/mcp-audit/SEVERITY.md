# SEVERITY.md — default severity weights for `mcp-audit` findings

Reference table mapping every finding type the [`mcp-audit` skill](SKILL.md)
can produce to its **default severity bucket**. The bucket meanings are
defined in [`docs/disclosure-sop.md`](../../docs/disclosure-sop.md#1-severity-rubric);
the disclosure channel that follows from each bucket is in the same SOP's
decision tree.

When the audit produces a case-study under `docs/case-studies/`, every
finding row carries one `finding_id` from the table below. Auditors **may
deviate** from the default if context warrants (a tautological docstring on
a tool that handles credentials is not Info, for example) — but deviations
must be called out in the case-study finding's "Why this bucket" line, not
silently applied.

---

## How to use this table

1. Audit produces a finding (e.g., "tool `site_health` returns raw dict").
2. Match the finding to the closest row by `category` + `description`.
3. Assign that row's `finding_id` and `default_severity` to the finding.
4. If you need to deviate (rare), record the override and the reason in
   the case-study finding block.

The 33 rows below cover the categories the audit's six sections naturally
produce, plus security-adjacent findings that surface when the audit is
applied to an *external* server (where the auditor has no prior trust
context with the maintainer).

## Override conventions

If you assign a non-default severity to a finding, prefix the
case-study's "Why this bucket" line with `OVERRIDE:` and one of:

- `OVERRIDE: escalated` — finding hits a higher bucket than default.
  Example: `DISC-001` (Info) escalated to Low because the tautological
  docstring is on `auth_token_create` and the tool is regularly
  miscalled by agents.
- `OVERRIDE: de-escalated` — finding hits a lower bucket than default.
  Example: `SEC-006` (High) de-escalated to Medium because the mutation
  tool is behind a documented `dev-mode` flag and the package README
  explicitly warns against production use.

Do not de-escalate `SEC-001`, `SEC-002`, or `SEC-003` (the three
Criticals). Those buckets are floors, not defaults.

---

## Adding a new row

When an audit surfaces a finding type that does not match any existing
row, propose a new row in the case-study's §6 ("Reusable patterns") and
open a follow-up PR against this file. Keep `finding_id` zero-padded and
contiguous within its category prefix.

The 33-row count is not sacred — the table is expected to grow as more
external audits land. The constraint is that every finding in every
case-study cites a row by `finding_id`, so coverage must precede usage.
